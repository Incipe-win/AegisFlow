# AegisFlow 接口文档

本文档是 `openapi/openapi.yaml` 的配套说明文档，用于帮助阅读接口语义、演示方式和前后端联调重点。真正的接口契约以 OpenAPI YAML 为准。

## 接口总览

| 场景 | 方法 | 路径 | 说明 |
| --- | --- | --- | --- |
| 智能问答 | `POST` | `/api/v1/chat` | 返回完整回答，支持多轮会话记忆 |
| 流式问答 | `POST` | `/api/v1/chat/stream` | 通过 SSE 持续输出回答片段 |
| AI 运维诊断 | `POST` | `/api/v1/ops/diagnose` | 输入告警上下文，返回排障分析和工具调用轨迹 |
| 文档上传 | `POST` | `/api/v1/knowledge/files` | 上传知识库文件 |
| 文档索引 | `POST` | `/api/v1/knowledge/index` | 执行分块和索引 |
| 运行详情 | `GET` | `/api/v1/runs/{runId}` | 查询一次对话或诊断的完整结果 |

## 旧接口映射

为了和早期设计稿保持语义一致，旧路径和新路径的映射关系如下：

| 旧路径 | 新路径 |
| --- | --- |
| `/api/chat` | `/api/v1/chat` |
| `/api/chat_stream` | `/api/v1/chat/stream` |
| `/api/ai_ops` | `/api/v1/ops/diagnose` |
| `/api/upload` | `/api/v1/knowledge/files` |

## 1. 智能问答接口

### 请求

`POST /api/v1/chat`

```json
{
  "sessionId": "session-001",
  "question": "支付服务 CPU 告警应该怎么排查？",
  "topK": 3
}
```

### 响应

```json
{
  "message": "OK",
  "data": {
    "sessionId": "session-001",
    "answer": "根据知识库和会话上下文，建议先检查 payment-service 的 CPU 占用、错误日志和最近发布记录。",
    "references": [
      {
        "documentId": "doc-seed-payment",
        "title": "payment-service 故障处理手册",
        "excerpt": "CPU 突增通常需要先核对最近 30 分钟的流量变化和慢 SQL。",
        "score": 0.91
      }
    ],
    "suggestions": [
      "继续追问当前告警的可能根因",
      "切换到 AI 运维页面执行自动诊断"
    ]
  }
}
```

## 2. 流式问答接口

### 请求

`POST /api/v1/chat/stream`

```json
{
  "sessionId": "session-001",
  "question": "帮我总结一下排查步骤",
  "topK": 3
}
```

### SSE 事件

- `connected`: 连接建立
- `message`: 模型逐段输出内容
- `reference`: 返回本次命中的知识片段
- `done`: 流结束
- `error`: 异常事件

### SSE 示例

```text
id: 1742970000
event: connected
data: {"status":"connected","sessionId":"session-001"}

id: 1742970001
event: message
data: 根据当前知识库内容，建议先检查 CPU 和错误日志。

id: 1742970002
event: reference
data: {"title":"payment-service 故障处理手册","score":0.91}

id: 1742970003
event: done
data: {"status":"completed"}
```

## 3. AI 运维诊断接口

### 请求

`POST /api/v1/ops/diagnose`

```json
{
  "alertTitle": "payment-service cpu usage high",
  "serviceName": "payment-service",
  "severity": "critical",
  "summary": "支付服务 CPU 连续 5 分钟高于 90%，错误率上升。",
  "sessionId": "ops-session-001"
}
```

### 响应

```json
{
  "message": "OK",
  "data": {
    "runId": "run-001",
    "result": "初步判断为流量上升叠加慢 SQL 导致 CPU 抬升，建议先限流并检查最近数据库查询。",
    "detail": [
      "读取告警上下文并提取服务名、严重级别和摘要",
      "从知识库检索 payment-service 的历史处理手册",
      "调用 Prometheus 工具查询 CPU、错误率和请求量趋势",
      "调用日志工具检查 error 关键字和超时信息",
      "调用 MySQL 工具检查最近慢查询摘要"
    ],
    "references": [
      {
        "documentId": "doc-seed-payment",
        "title": "payment-service 故障处理手册",
        "excerpt": "CPU 突增通常需要先核对最近 30 分钟的流量变化和慢 SQL。",
        "score": 0.91
      }
    ],
    "toolCalls": [
      {
        "name": "prometheus.query",
        "input": "cpu_usage{service=\"payment-service\"}",
        "output": "5m avg=93%, errorRate=4.8%, traffic +22%",
        "status": "success"
      }
    ]
  }
}
```

## 4. 文件上传接口

### 请求

`POST /api/v1/knowledge/files`

请求类型为 `multipart/form-data`，字段名为 `file`。

```bash
curl -X POST http://localhost:6872/api/v1/knowledge/files \
  -F "file=@README.md"
```

### 响应

```json
{
  "message": "OK",
  "data": {
    "documentId": "doc-uploaded-001",
    "fileName": "README.md",
    "filePath": "apps/api/resource/uploads/README.md",
    "fileSize": 1024,
    "status": "uploaded"
  }
}
```

## 5. 文档索引接口

```json
{
  "documentIds": ["doc-uploaded-001"],
  "chunkSize": 220,
  "overlap": 40,
  "topK": 3
}
```

```json
{

  "data": {
    "indexedDocuments": 1,
    "indexedChunks": 4,
    "chunkSize": 220,
    "overlap": 40,
    "topK": 3
  }
}
```

## 6. 运行详情接口

### 请求

`GET /api/v1/runs/{runId}`

### 响应

返回结构与诊断/对话运行保存的结果一致，可用于页面回显和运行历史查看。

## 联调建议

- 前端始终以 `openapi/openapi.yaml` 生成的类型为准。
- `chat/stream` 使用 `fetch` 读取 `text/event-stream` 数据流，不使用 `EventSource`。
- 本地演示建议先执行一次知识索引，再进行问答和 AI 运维诊断。
